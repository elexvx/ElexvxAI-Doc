## 目标
- 用 Astro 搭建一个可部署的文档网站，支持持续写作、结构化导航、良好 SEO 与可维护的主题样式。
- 本计划阶段的产出：把“最终需求”定下来（必做/可选、内容组织、信息架构、部署与运维约束），为后续实现提供明确边界。

## 已确认需求（当前定稿）
- 站点类型：对外产品文档
- 部署平台：Vercel
- 内容流程：纯 Git 协作（Markdown/MDX + PR 审核）
- 内容规模：100–300 篇（需要元数据规范与可扩展信息架构）
- 搜索：本地搜索（构建产物内完成索引，不依赖外部服务）
- 版本化：主版本策略（如 v1/v2/v3），需要版本切换
- 视觉参考：
  - https://developers.openai.com/codex/cloud
  - https://antigravity.google/

1. 语言：仅中文还是中英双语？
2. 域名：是否使用自定义域名（以及是否需要 /docs 作为子路径）？

## 建议的需求范围（默认方案，可调整）
### 必做（MVP）
- 文档内容：Markdown（可选 MDX）驱动，支持 Frontmatter 元数据（title、description、sidebar 等）。
- 信息架构：多级侧边栏 + 顶部导航 + 面包屑。
- 目录（TOC）：自动生成，支持标题锚点跳转。
- 搜索：站内全文搜索（优先本地索引方案，避免外部依赖）。
- 主题：亮/暗色切换；响应式；代码高亮；图片/表格排版友好。
- SEO：基础 meta、OpenGraph、站点地图 sitemap、规范化链接。
- 404/重定向：基础 404 页面；可选重定向配置。

### 可选（按优先级）
- i18n：中英双语与语言切换；同一路径策略（/zh、/en）或默认语言策略。
- 版本化文档：/v1、/v2；版本切换下拉。（本项目当前已纳入 MVP）
- 多产品线：按产品/模块分区（/product-a、/product-b）。
- 编辑体验：文档首页/标签/最近更新；贡献指南；编辑此页链接。
- 访问控制：基础鉴权（仅在需要时，且取决于部署平台能力）。
- 分析与反馈：站点统计、页面反馈（有用/无用）、评论（谨慎）。

## 内容与路由策略（建议）
- 使用 Astro Content Collections 统一管理文档集合与 schema 校验。
- 侧边栏由“自动扫描 + 手工排序配置”结合：避免纯自动导致结构失控。
- 约定每篇文档最少元数据：title、description、sidebar.position（或 weight）。

## 最终需求草案（可执行的 MVP 定义）
### 1) 路由与版本规则
- 默认版本：/docs 指向最新主版本（例如 /docs/v2）
- 版本入口：/docs/v1、/docs/v2（可扩展到 v3）
- 文档页面：/docs/<version>/<...slug>
- 版本切换：全站固定入口，切换后保持尽量相同的 slug（若不存在则落到该版本的文档首页）

### 2) 导航与信息架构
- 顶部导航：Docs、Changelog（可选）、GitHub（可选）、搜索入口
- 侧边栏：按版本独立（v1/v2 可不同），支持 2–3 级层级
- 文档页结构：标题区 + 描述 + TOC + 正文 + 上/下一篇 + 编辑此页（可选）

### 3) 搜索（本地）
- 优先采用“构建后生成索引”的方案（例如 Pagefind 类方案），提供：
  - 快捷键唤起（/ 或 Cmd/Ctrl+K 二选一）
  - 结果高亮与摘要
  - 过滤维度：版本（必需），模块（可选，取决于元数据）

### 4) 视觉与交互（基于参考站点收敛为一致风格）
- 基调：偏 OpenAI Developers 的清爽信息密度 + 适度动效（避免 antigravity 的强叙事交互影响阅读）
- 主题：亮/暗色；可配置品牌主色；排版对长文阅读友好
- 代码：高亮 + 复制按钮（MVP）；行号（可选）

### 5) SEO 与可访问性
- 每页：title/description、OpenGraph、规范化链接
- 站点：sitemap、robots、语义化结构与可访问性（键盘可达）

### 6) 交付与验收
- Vercel 可一键部署，PR 自动预览
- 任意文档可通过元数据进入正确侧边栏位置
- 搜索可在 100–300 篇规模下稳定可用
- v1/v2 版本切换与默认版本跳转符合预期

## 视觉与交互（建议）
- 参考风格：简洁、偏技术文档；注重可读性与可检索性。
- 代码块：行号（可选）、复制按钮（可选）。

## 交付物与验收标准（用于“最终需求”）
- 功能清单：明确哪些在 MVP，哪些延期。
- 导航结构：侧边栏目录树（到二级/三级）。
- 设计基调：颜色、字体、间距、组件风格的统一原则。
- 部署路径：确定部署平台、域名、构建命令与输出目录。

## 风险与约束
- 若需要私有访问控制，需提前确定部署平台与鉴权方式（否则后期会大改）。
- 若需要多语言/多版本，建议在一开始定好 URL 规则与内容组织。
